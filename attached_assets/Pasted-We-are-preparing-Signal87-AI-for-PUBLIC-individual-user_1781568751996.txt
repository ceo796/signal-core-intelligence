We are preparing Signal87 AI for PUBLIC individual-user deployment tonight.

This is now the deployment goal:

Public users should be able to visit the site, sign up or sign in using Google OAuth, including Gmail and Google Workspace accounts, enter the app, upload documents, chat with those documents, generate briefs, and only see/use their own documents.

This is not an internal allowlist deployment anymore.

Critical rules:

* Do not add Stripe.
* Do not add teams.
* Do not add workspaces.
* Do not add invite flows.
* Do not redesign the app.
* Do not change the core intelligence behavior.
* Do not change model routing.
* Do not change prompt behavior.
* Do not change citation formatting.
* Do not change brief formatting.
* Do not touch unrelated UI.
* Do not do broad refactors.
* This is a narrow public-access, authentication, and per-user document-isolation checkpoint.

Primary objective:
Make Signal87 safe for public individual-user access by ensuring every signed-in user has their own isolated document library.

Current known state:

* Public home page is accessible.
* Sign-in page exists and is branded.
* Clerk auth has been added.
* Private routes are protected.
* `/api/healthz` is public.
* `/api/documents` returns 401 when unauthenticated.
* `/api/brief` returns 401 when unauthenticated.
* There is currently a known shared-document-library limitation because documents do not yet have per-user ownership.
* That limitation must be fixed before public launch.

Required work:

1. Public signup/sign-in

Enable public sign-up and sign-in.

Users must be able to:

* Sign up using Google OAuth.
* Sign in using Google OAuth.
* Use Gmail accounts.
* Use Google Workspace accounts.
* Return later and access the same account.

Important:
Do not build a separate “Gmail login” system. Gmail login should be handled through the existing Google OAuth / “Continue with Google” flow.

Confirm:

* Sign-in page shows Google OAuth clearly.
* Sign-up page supports Google OAuth.
* New Google users can create accounts.
* Existing Google users can sign back in.
* Production Clerk environment variables are used for the live deployment.
* Redirect/callback behavior works on the deployed production URL.

2. Disable approved-user gate for public launch

The approved-email gate should not block public users.

Required:

* Do not require `VITE_APPROVED_EMAILS` for normal public access.
* Public users who sign up should be able to enter the app.
* Keep `/pending-access` available for future use, but do not send normal public users there unless a future gate is explicitly enabled.
* Make sure no allowlist setting accidentally blocks public users.

3. Add per-user document ownership

Add true document ownership using Clerk user IDs.

Required:

* Add an ownership column to the documents table.
* Preferred column name: `ownerUserId` or `clerkUserId`.
* Every newly uploaded document must be saved with the authenticated Clerk user ID.
* Reject upload/document creation if there is no authenticated Clerk user.
* Do not delete existing documents.
* Do not break current document IDs.

4. Handle existing ownerless documents

Existing documents without an owner must not appear to new public users.

Required:

* Do not delete existing documents.
* Ownerless documents should not be visible to normal public users.
* Either assign existing ownerless documents to a single admin/test Clerk user or hide ownerless documents from normal users.
* Choose the safest implementation with the least risk.
* Document exactly which approach was used.

5. Protect document list

Update document list behavior.

Required:

* `GET /api/documents` must only return documents where the ownership column equals the current authenticated Clerk user ID.
* Do not return all documents.
* Do not return ownerless documents to normal users.
* Confirm a new user starts with an empty document library.

6. Protect document detail

Update all document detail routes.

Required:

* A user can only access document detail for documents they own.
* Direct URL access to another user’s document must fail.
* Return 404 or 403 when a user requests a document they do not own.
* Apply this to every route that loads document detail, metadata, extracted text, summary, chunks, citations, or original file references.

7. Protect upload

Update upload behavior.

Required:

* Upload must require authentication.
* Upload must save the current Clerk user ID on the document.
* Uploaded documents must appear only in the uploading user’s document library.
* Another signed-in user must not see the uploaded document.

8. Protect chat and retrieval

This is critical.

Required:

* Chat retrieval must only search documents owned by the current authenticated user.
* Do not allow global retrieval across all documents.
* If document IDs are passed from the frontend, validate each document ID belongs to the current user before using it.
* If no valid owned documents are available, return a safe response.
* Do not let User B chat against User A’s document by guessing or passing User A’s document ID.
* Citations must only come from the current user’s owned documents.

9. Protect brief generation

Update brief generation ownership checks.

Required:

* Brief generation must only work with documents owned by the current user.
* Any selected document IDs must be validated against the current Clerk user ID.
* If a user requests a document they do not own, reject the request.
* Multi-document comparison must only compare documents owned by the same current user.
* Existing brief formatting and citation behavior should remain unchanged.

10. Protect derived document operations

Apply owner checks to every endpoint or function involving documents.

This includes:

* document detail
* document chunks
* document summaries
* document analysis
* document citations
* document chat
* document comparison
* brief generation
* download original
* PDF/original preview routes
* delete document
* reindex document
* upload
* any endpoint accepting `documentId`
* any endpoint accepting `documentIds`

Rule:
No document operation should proceed unless the document belongs to the authenticated Clerk user.

13. User A / User B isolation QA

Create or simulate two separate public users.

Test User A:

* Signs up with Google/Gmail.
* Enters the app.
* Sees an empty document library or only documents owned by User A.
* Uploads a document.
* Sees the uploaded document.
* Opens document detail.
* Asks a chat question.
* Receives an answer grounded only in User A’s document.
* Sees citations.
* Generates a brief.

Test User B:

* Signs up with a different Google/Gmail account.
* Enters the app.
* Does not see User A’s document.
* Cannot access User A’s document by direct URL.
* Cannot use User A’s document ID in chat.
* Cannot generate a brief using User A’s document ID.
* Cannot download User A’s original file.
* Cannot delete User A’s document.
* Cannot reindex User A’s document.
* Uploads a separate document.
* Sees only User B’s document.

Also confirm:

* User A cannot see User B’s document.
* User A cannot access User B’s document by direct URL.
* User A and User B document libraries remain separate.
