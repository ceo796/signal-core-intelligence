Implement Signal87 Hybrid Cross-Document Agent v1.

This is a controlled product implementation. Do not perform a broad refactor. Do not redesign the app. Do not touch authentication, Clerk configuration, database schema, storage configuration, document ingestion, upload, delete, re-indexing, PDF viewer, or existing working chat/brief endpoints except where explicitly necessary to reuse existing services.

Perform a narrow read-only inspection of the existing backend and frontend to identify:

* Existing brief generation route/service
* Existing document/chunk retrieval logic

* Existing frontend Ask, Documents, Briefs, Agents, or Home routes

Then report the intended files to change before implementing. If the current structure is unclear or there are competing implementations, stop and report.

Goal:
Create a clean v1 “Hybrid Agent” that can operate across the user’s documents.

Concept:
User asks a question → system identifies relevant user-owned documents → retrieves relevant chunks → synthesizes an answer → returns cited source documents/chunks → optionally recommends a next action.

Core requirements:

1. The agent must only use documents owned by the authenticated current user.
2. It must reuse existing document/chunk retrieval and AI provider logic wherever possible.
3. It must not duplicate large prompt logic if existing prompt utilities can be reused cleanly.
4. It must not introduce a new database, vector DB, external service, or embedding system unless one already exists.
5. It must be implemented as a clean service layer, not as a giant route handler.
6. It must preserve all existing single-document chat, multi-document comparison, brief generation, upload, re-index, download, and delete behavior.

Backend implementation:
Create a new endpoint only if no suitable endpoint already exists.

Preferred endpoint:
POST /api/agent/hybrid

Request shape:
{
“query”: “string”,
“documentIds”: [“optional array of document ids”],
“mode”: “auto | summarize | compare | extract | diligence”,
“maxDocuments”: 5,
“maxChunks”: 12
}

Response shape:
{
“answer”: “string”,
“mode”: “string”,
“documentsUsed”: [
{
“id”: “string or number”,
“name”: “string”
}
],
“citations”: [
{
“documentId”: “string or number”,
“documentName”: “string”,
“chunkId”: “string or number”,
“chunkIndex”: “number”,
“excerpt”: “string”
}
],
“trace”: {
“provider”: “string”,
“model”: “string”,
“documentsConsidered”: “number”,
“chunksConsidered”: “number”,
“latencyMs”: “number”
}
}

Retrieval behavior:

* If documentIds are provided, only search those documents, after confirming they belong to the current user.
* If documentIds are not provided, search across the current user’s available documents.
* Rank chunks using the simplest clean existing approach:
    * Prefer existing retrieval logic if already present.
    * Otherwise use safe lexical relevance over document names and chunk text.
* Limit the amount of text sent to the model.
* Do not include chunks from other users.
* Do not use placeholder documents.

Answer behavior:

* The agent must answer only from retrieved document context.
* If the documents do not contain enough information, it should say so clearly.
* It should cite the source documents/chunks used.
* It should not fabricate citations.
* It should not claim access to documents that were not retrieved.

Frontend implementation:
Add a simple Hybrid Agent interface in the existing app.

Preferred location:

* If an Agents page already exists, add the Hybrid Agent there.
* If no Agents page exists, add a simple protected route such as /agents/hybrid or use the existing Ask page if that is the current app convention.

UI requirements:

* Submit button.

Integration:

* The Home page ask bar may route to this agent only if that is cleaner than the existing Ask route.
* Do not break current Home navigation.
* Do not alter the Documents page layout in this task.
* Do not perform backend cleanup beyond what is necessary to add this feature cleanly.

Testing:
After implementation, verify:

1. Unauthenticated requests to /api/agent/hybrid are rejected.
2. Authenticated request with no documents returns a safe empty-state response.
3. Authenticated request across all current user documents returns an answer with citations.
4. Authenticated request with selected documentIds only uses those documents.
5. A user cannot access another user’s documents by passing their IDs.
6. Existing single-document chat still works.
7. Existing multi-document comparison still works.
8. Existing brief generation still works.
9. Existing upload, re-index, download original, and delete still work.
10. Frontend loads without console errors.
11. No database schema changes were made.
12. No auth/storage config was changed.
